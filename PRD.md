## **1\. Core Concept & Objectives**

### **The Problem: Context Pollution**

Standard AI assistants suffer from context pollution — when working on a complex, multi-domain project (e.g. a product with a frontend, ML pipeline, and security layer), a single chat window forces the AI to hold incompatible mental models simultaneously. Asking a Security question while in the middle of a React component discussion degrades the quality of both answers because the model cannot fully specialize.

### **The Solution: The Context Tree**

Context Tree is a non-linear AI reasoning engine implemented as a VS Code extension. It replaces the single chat window with a directed acyclic graph (DAG) of isolated Autonomous Agent Nodes. Each node is a fully self-contained AI agent with:

* Its own locked JSON Spec (system prompt, constraints, allowed tools, linked files)  
* Its own isolated chat history (the agent only sees its own context, never other nodes)  
* Its own Role identity (one of 6 specialist roles)  
* The ability to inherit context from parent nodes via directed graph edges

The graph is rendered with React Flow inside a VS Code Webview. The extension host bridges filesystem access (read/write) between the sandboxed webview and the local disk.

### **Design Principles**

1. Isolation First: No node can see another node's chat history unless explicitly merged  
2. Spec-Locked Execution: Agents operate within a hard-locked JSON spec generated by a Socratic interview. The spec is immutable until manually synced  
3. Directed Context Inheritance: Graph edges flow Parent → Child. A child node automatically inherits its parent's locked spec as an authoritative constraint in its system prompt  
4. Non-Linear Branching: Any node can be forked into a parallel branch, preserving history while allowing divergent exploration  
5. Compiler AI Merge: Two isolated nodes can be merged by a Compiler AI that performs conflict analysis and produces a unified spec \+ resolved code

### **3.4 Branch Forking (forkNode)**

Forking creates a full deep clone of a node (data \+ chatHistory), assigns a new UUID, resets its status to draft and specLocked: false, positions it 340px to the right, and re-attaches to the same parent. The fork is immediately selected.

forkNode(nodeId) {

 if (\!original) return undefined;  
 const newId \= uuidv4();

### **3.5 Rollback (rollbackNode)**

Truncates a node's chatHistory to \[0..messageIndex\] inclusive, effectively rewinding the conversation to any past point. This is a one-click time-machine for non-linear exploration.

### **4.2 Dynamic Socratic Builder (Draft Mode)**

When a node's metadata.status \=== "draft", the AI operates in Socratic interview mode. It asks the user one targeted domain question at a time to gather the information needed to generate a precise JSON spec.

Auto-initiation: When a draft node is selected and its chatHistory is empty, the interview is automatically kicked off with a silent bootstrap message:

*"I want to set up a node for the {role} role. Please begin the interview."*

Role-specific question sets — each role probes 5 distinct domains:

| Role | Domain Probes |
| :---- | :---- |
| Planner | Core objectives & success definition · Target audience & pain points · Feature prioritization (MVP vs nice-to-have) · Dependencies & risk vectors · Success metrics (KPIs/OKRs) |
| UI/UX Designer | Design system & component library · Key user flows & decision points · Accessibility (WCAG level) & device targets · Interaction patterns & state transitions · Handoff format (Figma tokens, CSS variables, Storybook) |
| Software Engineer | Tech stack constraints · Architecture pattern (monolith/microservices/serverless) · Data flow & entity relationships · External API/service contracts · Testing strategy & error handling |
| ML Engineer | Model type & problem framing · Training data availability, quality & labeling · Evaluation metrics (accuracy, latency, cost) · Deployment target (edge/cloud, batch/realtime) · MLOps (versioning, drift monitoring, retraining) |
| Security | Threat model & adversary profile · Compliance frameworks (GDPR, SOC2, HIPAA, PCI-DSS) · Auth/authz design (RBAC/ABAC, SSO/OIDC) · OWASP Top 10 surface analysis · Incident response & forensics SLA |
| QA | Test coverage targets & critical paths · Test pyramid (unit/integration/contract/e2e/perf) · Environment topology & data seeding strategy · Acceptance criteria format (BDD/Gherkin, sign-off) · CI trigger strategy & flaky test SLA |

Protocol rules (injected into every Socratic system prompt):

* Ask ONE highly specific, expert-level question at a time  
* Never generate code, plans, or specs during the interview  
* Ask meaningful follow-ups that probe deeper priorities and exact failure modes  
* Goal: gather enough to produce a precise, actionable spec

Spec generation — user clicks "📐 Lock Spec". The entire chatHistory (excluding system role messages) is submitted to the LLM with this prompt structure:

You are a JSON spec generator for the Context Tree system.  
Based on the interview transcript below, produce a strict JSON specification.  
Node Context:  
\- Role: {role}  
Interview Transcript:  
{chatHistory filtered to user/assistant, formatted as \[ROLE\]: content}  
Output ONLY a valid JSON object matching this schema (no markdown, no fences):  
{  
 "system\_prompt": "Comprehensive AI operating instructions synthesized from the interview",  
 "constraints": \["Each hard rule or limitation — one string per constraint"\],  
 "context\_splicing\_rules": "How this node's context combines with parent/sibling nodes",  
 "allowed\_tools": \["Specific tools or capabilities needed"\],  
 "linked\_files": \["Absolute file paths mentioned, or empty array"\],  
 "status": "active",  
 "specLocked": true  
}

The result is validated, merged into node.metadata, and specLocked is set to true. The node transitions to active status.

### **4.3 Active Node Agent Chat**

Once specLocked: true, the node operates as a full autonomous agent. The system prompt for active mode:

{metadata.system\_prompt}  
\#\# Filesystem Access  
You have direct access to the user's local filesystem via tool calls.  
When writing code: ALWAYS use \`create\_or\_edit\_local\_file\` — never output raw fenced code blocks.  
When inspecting existing code before editing: use \`read\_local\_file\` first.  
\#\# Inherited Parent Context (if parent nodes exist via graph edges)  
\#\#\# Parent N: "{label}" ({role})  
\- System Prompt: {parent.system\_prompt}  
\- Constraints: {parent.constraints}  
\- Context Splicing Rules: {parent.context\_splicing\_rules}  
Your output MUST be compatible with all parent constraints above.  
\#\# This Node's Locked Spec  
\- Constraints: {metadata.constraints}  
\- Context Splicing: {metadata.context\_splicing\_rules}  
\- Allowed Tools: {metadata.allowed\_tools}  
\- Linked Files: {metadata.linked\_files}  
Stay strictly within this spec. Never violate any constraint.

Tool calling loop: The LLM client runs a recursive streaming loop (MAX\_TOOL\_ROUNDS \= 8). If the model requests a tool call, the loop:

1. Appends the assistant message with tool\_calls  
2. Executes each tool via ToolExecutor (bridged through the extension host using REQUEST\_FILE\_READ / REQUEST\_FILE\_WRITE messages)  
3. Appends the tool results  
4. Continues to the next round until finish\_reason \!== "tool\_calls" or the round limit is reached

This loop works identically for both OpenAI (via SSE streaming \+ tool\_calls deltas) and Anthropic (via SSE streaming \+ tool\_use / tool\_result content blocks).

Parent context resolution: Before every active-node chat message, the store is queried for all edges where edge.target \=== currentNodeId. The source nodes' locked specs are collected and injected as the "Inherited Parent Context" section in the system prompt.

### **4.4 Context Sync ("Sync Context to Spec")**

Problem: After a node becomes active and the user continues chatting, new decisions are made that contradict or supersede the original locked spec. The spec becomes stale.

Solution: A "🔄 Sync Context to Spec" button appears on active nodes. When clicked:

1. The full chatHistory (excluding system messages) is assembled with sequential message numbers  
2. The current spec JSON is included as the "baseline to update"  
3. This prompt is submitted to the LLM:

You are a JSON spec synchronizer for the Context Tree system.  
Rewrite the JSON metadata spec for this node based on its ENTIRE conversation history.  
CRITICAL WEIGHTING RULE: Give SIGNIFICANTLY HIGHER WEIGHT to the most recent messages.  
If recent messages contradict, override, or refine older constraints or decisions,  
the NEWER information ALWAYS wins. Old constraints must be replaced when superseded.  
Node Context:  
\- Role: {role}  
Full Conversation History (oldest → newest):  
\[MSG 1\]\[USER\]: ...  
\[MSG 2\]\[ASSISTANT\]: ...  
\[MSG N\]\[USER\]: ...  ← these have highest weight  
Current Spec (baseline):  
{JSON.stringify(current metadata fields)}  
Output ONLY a valid JSON object (no markdown, no fences):  
{  
 "system\_prompt": "...",  
 "constraints": \[...\],  
 "context\_splicing\_rules": "...",  
 "allowed\_tools": \[...\],  
 "linked\_files": \[...\],  
 "status": "active",  
 "specLocked": true  
}

4. The returned JSON is validated and merged into node.metadata via updateNodeData(nodeId, { metadata: newSpec })  
5. The node's active status is preserved — it does NOT revert to draft  
6. A system message is appended to chatHistory: "🔄 Spec synced — metadata rewritten from full chat history (recent messages take precedence)."

Two active nodes can be merged. The user enters Merge Mode (toolbar button), clicks two nodes, then triggers "Run Compiler AI →" in the Merge panel.

Compiler AI prompt sends both nodes' full metadata specs \+ chat histories and demands this exact JSON response:

   "constraints": \["Consolidated hard constraints from both nodes"\],

Strict schema validation is enforced before the merge node is created. If the LLM returns invalid JSON or an incomplete schema, an error is thrown with the raw response for debugging.

Its chat history is the concatenation of A's history, a system separator, B's history, and the compiler's conflict analysis \+ resolved code. ## **5\. Advanced Requirements for Ralph (The Upgrade Mandate)**

### **5.1 Current MVP Limitations to Overcome**

The current MVP has these known gaps that Ralph must address in the rebuilt version:

1. Passive Tool Access: Agents currently only use filesystem tools when the user explicitly asks them to write or read a file in chat. They do not proactively inspect linked\_files.  
2. No Auto-Validation: Agents write code but do not verify it compiles, passes tests, or matches constraints.  
3. Single LLM Provider at a Time: The config stores one provider. Multi-model routing (e.g., use Claude for reasoning, GPT-4o for code gen) is not implemented.  
4. No Real-Time Graph Updates: When an agent writes a file via tool call, the linked\_files array in the spec is not automatically updated to track that file.

### **5.2 Mandatory Agentic Tool Calling for Every Node**

RALPH MUST implement the following in the new version:

Every active node agent MUST behave as a fully autonomous coding agent, not just a chatbot. Specifically:

On activation (when a node transitions from draft → active), the agent MUST:

1. Automatically read all files listed in metadata.linked\_files using read\_local\_file  
2. Internalize their contents as live context before responding to the first user message  
3. If linked\_files is empty, prompt the user for relevant files and add them to the spec

On every response, the agent MUST:

1. Prefer writing code directly to the filesystem via create\_or\_edit\_local\_file over outputting code blocks in the chat  
2. After writing any file, automatically attempt to validate its own output (e.g., for TypeScript: check for syntax errors; for Python: attempt an import; for tests: run them)  
3. Report the validation result as part of its response

On spec sync, the sync operation MUST also:

1. Scan the chatHistory for any file paths mentioned by the agent (regex: /\[A-Za-z\]:\[\\\\\\/\]\[\\w\\\\/\\-\\.\]+|\\/\[\\w\\/\\-\\.\]+\\.\\w+/g)  
2. Auto-add any discovered paths to linked\_files that aren't already present  
3. Include this updated file list in the synced spec

### **5.3 Multi-Agent Coordination**

In the rebuilt version, Ralph must implement a "Broadcast" mechanism:

### **5.4 Tool Executor Architecture**

     case "read\_local\_file":  
       return this.readFile(args.path as string);  
     case "create\_or\_edit\_local\_file":

## **7\. Exact LLM Client Implementation Details**

The entire LLM communication layer lives in utils/llmClient.ts. Ralph must implement all of the following:

### **7.1 Streaming Architecture**

Both OpenAI and Anthropic use SSE (Server-Sent Events) streaming. The client reads the response body as a ReadableStream, decodes chunks via TextDecoder, splits on \\n, and processes data: {...} lines.

OpenAI SSE: Each delta contains choices\[0\].delta.content (text) or choices\[0\].delta.tool\_calls (array of partial tool call deltas assembled by index key). Stop signal: data: \[DONE\].

Anthropic SSE: Events are typed (content\_block\_start, content\_block\_delta, message\_delta). Text blocks yield text\_delta. Tool use blocks yield input\_json\_delta (partial JSON accumulated per block index). Stop signal: message\_delta.stop\_reason.

### **7.2 Tool Calling Loop**

const MAX\_TOOL\_ROUNDS \= 8;  
async function runOpenAI(messages, config, tools, toolExecutor, onChunk, round): Promise\<string\> {  
 if (round \>= MAX\_TOOL\_ROUNDS) return "";  
 const { text, toolCalls, finishReason } \= await doOpenAIStream(messages, config, tools, onChunk);  
 if (finishReason \!== "tool\_calls" || toolCalls.length \=== 0 || \!toolExecutor) return text;  
 *// Append assistant message with tool\_calls*  
 const next \= \[...messages, {  
   role: "assistant", content: text || null,  
   tool\_calls: toolCalls.map(tc \=\> ({ id: tc.id, type: "function", function: { name: tc.name, arguments: tc.arguments } }))  
 }\];  
 *// Execute each tool, append results*  
 for (const tc of toolCalls) {  
   const args \= JSON.parse(tc.arguments || "{}");  
   const result \= await toolExecutor(tc.name, args).catch(err \=\> \`Tool error: ${err}\`);  
   next.push({ role: "tool", tool\_call\_id: tc.id, content: result });  
 }  
 const continuation \= await runOpenAI(next, config, tools, toolExecutor, onChunk, round \+ 1);  
 return (text ? text \+ "\\n" : "") \+ continuation;  
}

Anthropic uses equivalent logic but with tool\_use / tool\_result content blocks in the message array format.

### **7.3 Non-Streaming (callLLM)**

Used for spec generation, spec sync, and Compiler AI merge (all require validated JSON responses):

* OpenAI: response\_format: { type: "json\_object" } is set to force JSON mode  
* Anthropic: No JSON mode; the prompt must be explicit, and the response is stripped of any accidental markdown fences before parsing

## **8\. Constraints Ralph Must Never Violate**

1. No window.confirm(): VS Code webviews suppress native dialogs. All confirmations must use inline React state (the two-step rollback pattern: set pendingId state → show confirm/cancel buttons inline).  
2. No NodeLevel: This field is permanently removed. Do not reintroduce it under any name.  
3. persistDb() on every mutation: Every store action that modifies nodes or edges must call get().persistDb() at the end to sync to disk.  
4. Arrow on every edge: markerEnd: { type: MarkerType.ArrowClosed } must be present on every ContextEdge, without exception. The visual parent→child direction is a core UX requirement.  
5. Spec immutability in draft mode: A draft node's chatHistory is append-only during the Socratic interview. The spec is only generated once, atomically, when the user clicks "Lock Spec". There is no partial spec writing.  
6. Bridge response ID matching: Every REQUEST\_FILE\_READ and REQUEST\_FILE\_WRITE message includes a UUID id. The extension host echoes this id back in the response. The webview uses a Map\<string, resolver\> (the pendingBridgeReqs ref) to match async responses. Bridge requests have a 30-second timeout that rejects with an error.  
7. API key never reaches extension host: LLM calls are made directly from the webview via fetch() to https://api.openai.com or https://api.anthropic.com. The extension host's CSP (connect-src https:) permits this. The API key is stored in localStorage under key "context-tree:llm-config" and never sent via postMessage.
